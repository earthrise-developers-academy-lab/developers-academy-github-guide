# Issues and deliverables

Use an issue for a bounded deliverable, task, bug, feature, research question, decision, documentation requirement, or blocker.

A deliverable issue should identify:

- **Objective** — what result is needed
- **Required artifact** — what must exist
- **Working location** — where current work is happening
- **Acceptance criteria** — what must be true before completion
- **Reviewer** — who confirms acceptance
- **Completion evidence** — links, results, or approvals

## Claim an issue

1. Confirm it is not already assigned.
2. Assign yourself or ask a maintainer.
3. Move the Project status to **In progress**.
4. Comment with your intended first action.

```markdown
### Starting work

I will review the existing requirements and create the first user-guide outline.

Working document: <Google Doc URL>

I expect to request review after the installation, primary workflow, and known-limitations sections are drafted.
```

## Report a blocker

```markdown
### Blocked

**Blocker**
I do not have access to the source dataset.

**Impact**
Validation cannot begin.

**Needed**
A maintainer must confirm the approved data location or grant access.
```

Move the issue to **Blocked**.
